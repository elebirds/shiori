# 可观测性面试问答

## 一、方案动机

### Q1：为什么要做这次可观测性改造？

答：

因为我原来只有两类证据：压测报告和应用日志。它们能说明系统有没有慢、有没有报错，但不能很好回答“瓶颈到底在哪一层”。

所以我补了一套最小可用闭环：

1. 应用暴露 Prometheus 指标。
2. Prometheus 负责采集。
3. Grafana 用总览面板展示关键指标。

这样我在压测时不仅能看到“慢了”，还能继续判断是同步链路、异步链路还是数据库资源先出问题。

### Q2：为什么不直接看日志？

答：

日志更适合查单次异常和具体报错，不适合持续观察吞吐、延迟分布、Kafka 积压和连接池水位。定位性能瓶颈，时序指标的价值比日志更高。

更准确一点说，日志回答的是“发生了什么”，指标回答的是“整体趋势怎么变化、瓶颈出现在哪”。

### Q3：为什么这次不一上来做全链路追踪？

答：

因为这次目标是“最高性价比地补最小闭环”。Actuator + Prometheus + Grafana 接入成本低、改造小，已经足够支撑压测定位和面试讲述。

分布式追踪很有价值，但它更适合下一阶段增强，不是这次最小闭环的必要条件。

## 二、方案设计

### Q4：你这次具体接了什么？

答：

我做的是 `Spring Boot Actuator + Prometheus + Grafana` 这条链路：

1. 多个 Spring Boot 服务暴露 `/actuator/prometheus`
2. Prometheus 按 job 定时抓取
3. Grafana 预置 `Shiori Overview` 总览面板展示关键面板

这三个部分仓库里都有真实落点，不是只写在文档里。

### Q5：你为什么选这几类指标？

答：

我不是追求大而全，而是优先选最能解释瓶颈的指标：

1. 订单创建 QPS：看业务吞吐有没有跟上。
2. 库存扣减延迟：看同步交易关键路径有没有变慢。
3. Kafka 消费 lag：看异步消费有没有积压。
4. Hikari 活跃连接数：看数据库连接池是否接近上限。
5. HTTP QPS / p95 / 5xx：看总体流量、延迟和错误情况。

这套组合已经足够支撑一轮压测定位。

### Q6：订单创建 QPS 为什么不用新埋点？

答：

我复用了订单状态迁移指标 `shiori_order_transition_total{from="NEW",to="UNPAID"}`。因为在当前业务里，订单创建成功后一定会进入 `UNPAID`，所以它能作为订单创建吞吐的近似观测。

这不是最理想的独立指标，但它复用现有语义，成本低，而且足够准确。

## 三、关键指标与实现

### Q7：库存扣减延迟是怎么做的？

答：

我在 `ProductStockService.deduct(...)` 上埋了 Micrometer `Timer`，指标名是 `shiori_product_stock_deduct_latency_seconds`。

我还按结果打了标签：

1. `success`
2. `idempotent_success`
3. `stock_not_enough`
4. `error`

这样不仅能看总体延迟，还能区分不同业务结果下的耗时分布。

### Q8：为什么库存扣减更适合看 p95，不看平均值？

答：

因为平均值会掩盖尾延迟，而高并发场景真正暴露瓶颈的往往是尾部慢请求。看 p95 更容易看出数据库争用、锁等待或者某些下游抖动。

### Q9：Kafka 消费 lag 是怎么做的？

答：

我在 `WalletBalanceOutboxCdcConsumer` 里，用消息的 `record.timestamp()` 和当前处理时间做差，然后通过 `OrderMetrics.recordKafkaConsumerLagSeconds(...)` 记录成 `shiori_order_kafka_consumer_lag_seconds`。

这个指标表达的是：

1. 消息写入 Kafka 后
2. 到当前消费者真正开始处理时
3. 已经滞后了多少秒

### Q10：这是不是最标准的 consumer group lag？

答：

不是 broker 侧最标准的 lag 指标。更准确地说，它是“消息年龄”或者“消费处理延迟”。

但它的优点是：

1. 接入成本低。
2. 不依赖额外 exporter。
3. 足够支撑本地排障和面试讲解。

如果后面继续做生产级可观测性，可以再补 broker/exporter 侧更标准的 lag 指标。

### Q11：为什么连接池活跃连接数也很重要？

答：

因为很多 Java 服务的性能问题最后都会表现成数据库连接资源紧张。看 `hikaricp_connections_active`，可以帮助我区分两类问题：

1. 延迟升高且连接池逼近上限，优先怀疑数据库或连接池。
2. 延迟升高但连接池并不高，说明瓶颈更可能在业务逻辑、下游调用或异步积压。

## 四、定位方法

### Q12：如果压测时延迟升高，你会怎么用这套面板定位？

答：

我的判断顺序是：

1. 先看 `HTTP p95 Latency` 和 `HTTP 5xx Ratio`，确认是不是整体变慢或报错。
2. 再看 `Order Create QPS`，判断吞吐有没有跟上压测流量。
3. 如果 `Stock Deduct p95 Latency` 上升，说明同步交易链路在变慢。
4. 如果 `Kafka Consumer Lag` 持续升高，说明异步消费在积压。
5. 如果 `Hikari Active Connections` 接近上限，说明数据库连接池或数据库资源可能成为瓶颈。

也就是说，这套面板不是为了“好看”，而是为了形成排障顺序。

### Q13：HTTP 指标和业务指标各自解决什么问题？

答：

HTTP 指标解决“哪个服务先出问题”，业务指标解决“为什么出问题”。

例如：

1. `HTTP p95` 告诉我系统慢了。
2. `Stock Deduct p95` 告诉我库存关键路径是否变慢。
3. `Kafka Consumer Lag` 告诉我异步链路是否已经堆住。
4. `Hikari Active Connections` 告诉我数据库资源是不是在告急。

## 五、验证范围与边界

### Q14：这次你验证到什么程度？

答：

这题一定要实话实说。当前我能确认的是：

1. Grafana 本体能启动。
2. Prometheus 本体能启动。
3. Grafana datasource provisioning 已加载。
4. `Shiori Overview` 总览面板已自动加载。
5. 关键指标、PromQL 和面板定义都已经落到仓库里。

但我不会夸大说“本地所有业务指标曲线都完整验证了”，因为当前还有本地环境问题影响抓取稳定性。

### Q15：具体是什么环境边界？

答：

当前本地多个业务服务容器因为 MySQL 旧数据卷密码不一致而反复重启，这会导致 Prometheus 对部分服务的 target 抓取不稳定，可能出现 `down/EOF`。

所以更准确的说法是：

> 可观测性基础设施和面板预置已经跑通，但完整业务曲线展示仍受本地环境问题影响。

### Q16：如果面试官问“这算不算真正做完可观测性”，你怎么答？

答：

如果按生产级标准看，这当然还不算完整可观测性，因为还缺：

1. 告警规则
2. 分布式追踪
3. 日志聚合
4. 更标准的 Kafka lag 观测

但如果按当前项目阶段目标看，这次已经把“压测之后能定位瓶颈”的最小闭环补上了，这是有明确价值的。
