# 网关关键写接口限流面试问答（GCRA 终版）

## 一、方案动机

### Q1：为什么要改原来的网关限流？

答：

因为原来是单机内存固定窗口计数，主要有两个问题：

1. 秒边界会临界突刺。
2. 多实例网关不共享计数，扩容后总放行量会被实例数放大。

所以我先把它升级成 Redis + Lua 的滑动窗口，后面又进一步演进成 Redis + Lua 的 GCRA。

### Q2：为什么不是一步到位直接上 GCRA？

答：

因为滑动窗口是对固定窗口问题最直接的修复，第一步先把精度和多实例一致性问题解决掉，风险最低、收益也最直观。

在第一步稳定之后，我再进一步把它演进成 GCRA，目的是降低 Redis 状态成本。

---

## 二、算法选型

### Q3：为什么终版从滑动窗口换成 GCRA？

答：

滑动窗口虽然精确，但典型实现要在 Redis 里维护窗口内请求日志，通常是 `ZSET`。  
这样做的代价是：

1. 每次请求都要清理旧记录
2. 需要维护窗口内的请求集合
3. 状态量和请求密度正相关

而 GCRA 只需要维护一个 `TAT`，也就是理论到达时间。  
它不记录“过去发生了多少请求”，而是记录“当前已经预支了多少未来额度”，所以 Redis 状态更轻。

### Q4：GCRA 和令牌桶是什么关系？

答：

它们本质上是在描述同一件事，只是表达方式不同：

1. 令牌桶看的是“我还剩多少额度”
2. GCRA 看的是“我已经欠了多少未来时间”

令牌桶更像余额模型，GCRA 更像时间债务模型。

### Q5：GCRA 里最重要的状态是什么？

答：

最重要的是 `TAT`，也就是 `Theoretical Arrival Time`。

可以把它理解成：

> 按当前已经消费掉的速率额度，下一次理论请求时间已经排到哪里了。

如果当前请求比允许时间来得更早，说明它在超前消费未来额度，就应该被拒绝。

## 三、实现方式

### Q6：你怎么在 Redis 里实现 GCRA？

答：

我是用 Redis + Lua 做的。每个“接口 + 身份”组合对应一个 key，比如：

```text
gateway:rate-limit:order_pay:uid:1001
```

值里不再存请求日志，而是只存一个 `TAT`。

Lua 脚本一次完成：

1. 读取旧 `TAT`
2. 计算当前请求是否过早到达
3. 如果允许，写回新的 `TAT`
4. 如果不允许，返回 `retryAfterMs`

### Q7：为什么一定要 Lua？

答：

因为如果把“读旧状态、算是否超限、写回新状态”拆到应用层，就会有并发竞态。

Lua 的价值在于把判断和更新变成 Redis 侧的一次原子操作。

### Q8：你还在 Redis 里用 `ZSET` 吗？

答：

终版不用了。  
滑动窗口阶段用的是 `ZSET`，GCRA 终版只用单值 `GET/SET` 维护 `TAT`。

这是两版最大的实现差异。

## 四、限流维度

### Q9：登录、下单、支付是怎么划分限流身份的？

答：

1. 登录按 IP 限流
2. 下单优先按用户 ID，缺失时退回 IP
3. 支付优先按用户 ID，缺失时退回 IP

这样做是为了兼顾安全性和公平性。

### Q10：为什么支付不按订单号限流？

答：

因为网关限流治理的是入口流量，不是订单幂等正确性。  
订单级重复支付保护还是要靠业务侧幂等、状态机和持久化校验，而不是靠网关限流替代。

## 五、异常与降级

### Q11：Redis 挂了怎么办？

答：

我没有让 Redis 成为关键写接口的硬依赖。当前策略是：

1. 正常走 Redis + Lua GCRA
2. Redis 脚本失败时，降级到本地固定窗口限流

### Q12：为什么不直接全部放行？

答：

因为这些都是关键写接口：

1. 登录入口完全放行会加大撞库和刷接口风险
2. 下单和支付入口完全放行，会让治理层在故障时彻底失效

我的取舍是：

1. 接受降级期间精度下降
2. 但不接受关键写接口在 Redis 故障时完全裸奔

---

## 六、可观测性

### Q13：你怎么知道线上有没有频繁走降级？

答：

我把限流指标拆成了四类：

1. `allow`
2. `blocked`
3. `degraded_allow`
4. `degraded_block`

这样可以直接看出：

1. 某个接口有没有被频繁限流
2. Redis 异常时是不是大量流量落到了降级路径

---

## 七、取舍与边界

### Q14：GCRA 的代价是什么？

答：

它的主要代价不是 Redis 成本，而是理解门槛更高：

1. 需要解释 `TAT`
2. 需要解释 `Emission Interval`
3. 需要解释 `Burst Tolerance`

所以我在对外讲项目时，会先说我经历了“固定窗口 -> 滑动窗口 -> GCRA”的演进，这样面试官更容易跟上。

### Q15：如果后面流量继续变大，你下一步怎么做？

答：

我会继续从两个方向优化：

1. 如果是单个热点身份极端集中，可以进一步细化接口分组和阈值
2. 如果是网关治理能力要平台化，可以把 GCRA 抽成通用限流组件，并补更完整的动态配置和灰度能力

但当前阶段，GCRA 已经比固定窗口和滑动窗口更适合作为关键写接口的长期方案。
