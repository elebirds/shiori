# 网关关键写接口限流面试问答（滑动窗口版）

## 一、方案动机

### Q1：为什么要改掉原来的网关限流？

答：

因为原来是单机内存固定窗口计数，虽然实现简单，但有两个明显问题：

1. 秒边界会临界突刺。
2. 多实例网关之间不共享计数，扩容后整体阈值会被放大。

所以我把它升级成了 Redis + Lua 的滑动窗口限流。

### Q2：你这次治理的是哪些接口？

答：

主要是关键写接口：

1. 登录
2. 下单
3. 支付

这些接口一旦被高频刷，会直接影响认证、防刷和交易链路，所以我优先从这里做限流。

---

## 二、算法设计

### Q3：为什么选滑动窗口，不是固定窗口？

答：

滑动窗口统计的是“最近一段时间内真实发生的请求数”，不会像固定窗口那样在自然秒边界放大流量。

举个例子，假设 1 秒最多 5 次：

1. 固定窗口在 `0.99s` 放 5 次、`1.01s` 还能再放 5 次。
2. 滑动窗口会看到前面那 5 次还在最近 1 秒里，所以会拒绝后续请求。

### Q4：你怎么在 Redis 里实现滑动窗口？

答：

我是用 `ZSET + Lua` 实现的：

1. key 维度是 `接口 + 身份`
2. `score` 是请求毫秒时间戳
3. `member` 是 `timestamp-nonce`

Lua 脚本一次做三件事：

1. 清理窗口外旧请求
2. 统计窗口内请求数
3. 决定是拦截还是写入当前请求

### Q5：为什么要用 Lua？

答：

因为如果不用 Lua，而是先查再写，就会有并发竞态：

1. 两个请求几乎同时查到“还没超限”
2. 然后都写进去
3. 最终会超过阈值

Lua 可以把这些动作变成 Redis 侧的一次原子执行。

---

## 三、限流维度

### Q6：登录、下单、支付都是按 IP 限流吗？

答：

不是。

1. 登录前还没有稳定用户身份，所以按 IP。
2. 下单和支付优先按用户 ID。
3. 如果用户 ID 缺失，再退回 IP。

这样既能保护接口，又能减少 NAT 场景下对正常用户的误伤。

### Q7：为什么支付要按用户 ID，不按订单号？

答：

因为这里治理的是“用户级刷接口”问题，不是订单级幂等问题。

支付正确性和重复支付保护，最终还是要靠业务侧幂等和状态机来保证；网关限流只是前置削峰和防刷。

---

## 四、异常与降级

### Q8：Redis 挂了怎么办？

答：

我没有选择“Redis 一挂就完全放行”，因为这些都是关键写接口。当前做法是：

1. 正常走 Redis + Lua 滑动窗口
2. Redis 脚本失败时，降级到本地固定窗口限流

这样即使降级期间精度变差，关键写入口也不会完全裸奔。

### Q9：为什么不直接失败？

答：

因为 Redis 在这里是治理组件，不应该直接成为业务可用性的硬依赖。  
如果 Redis 一抖动，登录、下单、支付全部失败，代价太大。

所以我选的是：

> 主路径追求精确，多实例统一；异常路径保底防刷，不让服务完全失去保护。

---

## 五、指标与可观测性

### Q10：你怎么知道线上有没有误伤或者频繁降级？

答：

我把网关限流指标分成了四类结果：

1. `allow`
2. `blocked`
3. `degraded_allow`
4. `degraded_block`

这样可以直接看：

1. 是否真的在命中限流
2. 有没有大量流量落到降级路径
3. 某个接口是不是限流阈值配置过紧

---

## 六、取舍与边界

### Q11：这版滑动窗口方案的代价是什么？

答：

主要代价是 Redis 成本更高：

1. 每次请求都要读写 `ZSET`
2. 窗口内请求越多，临时状态越多
3. 如果把它直接推广到全站所有高频接口，Redis 压力会明显增加

所以我把它先落在关键写接口，而不是全站所有流量入口。

### Q12：如果后面 QPS 再上来，你下一步会怎么做？

答：

如果关键写接口的吞吐继续抬高，我会考虑把滑动窗口进一步演进成 GCRA 或令牌桶一类更轻状态的限流算法，用更小的 Redis 状态成本来维持相近的限流效果。

## 七、完整回答模板

### 30 秒版本

> 我把网关关键写接口限流从单机固定窗口升级成了 Redis + Lua 的滑动窗口限流，主要覆盖登录、下单和支付。固定窗口的问题是秒边界会突刺，而且多实例计数不一致；滑动窗口会统计最近 1 秒内真实请求数，可以避免边界双倍放量。Lua 保证删旧数据、统计请求数和写入当前请求是原子操作。Redis 异常时，我没有直接全部放行，而是降级到本地固定窗口做兜底。

### 90 秒版本

> 这次我治理的是网关关键写接口，也就是登录、下单和支付。原来的方案是本地内存固定窗口，它能挡住一部分流量，但会在秒边界出现临界突刺，而且多实例下每台网关各算各的。  
> 后来我改成了 Redis + Lua 的滑动窗口限流。具体上，每个接口和身份组合对应一个 Redis ZSET，score 是毫秒时间戳，member 是时间戳加 nonce。每次请求会先清掉窗口外旧记录，再统计当前窗口内请求数，如果已经达到阈值就返回 429 和 retry-after，否则写入当前请求。  
> 登录按 IP 限流，下单和支付优先按用户 ID 限流。为了防止 Redis 故障时关键写接口完全失去保护，我还保留了本地固定窗口作为降级兜底，这样主路径足够精确，异常路径也有保底能力。
