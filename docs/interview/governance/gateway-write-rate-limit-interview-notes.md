# 网关关键写接口限流改造原理讲解（滑动窗口版）

## 1. 背景：为什么要改掉原来的限流

这次改造的对象是 `gateway-service` 里的关键写接口限流，入口在 `GatewayWriteOpsRateLimitFilter`，主要覆盖：

1. 登录 `POST /api/user/auth/login`
2. 下单 `POST /api/v2/order/orders`
3. 支付 `POST /api/v2/order/orders/{orderNo}/pay`

改造前的实现是“单机内存 + 每秒固定窗口计数”。这个方案能挡住一部分突发流量，但有两个明显问题：

1. **固定窗口临界突刺**：比如某个用户在 `00:00:00.999` 打满 5 次，又在 `00:00:01.001` 再打 5 次，两个自然秒桶都会放行，瞬间会穿透成 10 次。
2. **多实例不一致**：计数放在网关本地内存里，每台网关各算各的，横向扩容后限流上限会被实例数放大。

所以这次我把它升级成了：

> **Redis + Lua 的滑动窗口限流**，并保留本地固定窗口作为 Redis 异常时的降级兜底。

---

## 2. 改造目标

这次改造不是为了做全站统一网关治理平台，而是先把关键写接口限流做成一个“可以上线、可以解释、可以追问”的版本，目标有 4 个：

1. 明确限流算法，从固定窗口改成滑动窗口。
2. 把状态从单机内存搬到 Redis，保证多实例统一口径。
3. 用 Lua 保证“删旧数据、统计窗口、写入当前请求”是原子操作。
4. Redis 异常时自动降级到本地固定窗口，避免关键写接口完全裸奔。

---

## 3. 方案总览

### 3.1 限流维度

三个接口不是用同一个维度限流：

1. 登录接口按 **IP** 限流。
2. 下单接口优先按 **用户 ID** 限流；如果用户标识缺失，再退回 IP。
3. 支付接口优先按 **用户 ID** 限流；如果用户标识缺失，再退回 IP。

这样做的原因是：

1. 登录前通常还没有稳定用户身份，只能先按 IP 限流。
2. 已登录写接口按用户维度更公平，不容易把同一 NAT 出口下的正常用户一起误伤。

### 3.2 滑动窗口怎么落到 Redis

每个“接口 + 身份”组合对应一个 Redis key，例如：

```text
gateway:rate-limit:order_pay:uid:1001
gateway:rate-limit:login:ip:10.0.0.1
```

值类型用的是 `ZSET`，每次请求写一条“请求时间戳”记录：

1. `score` 用毫秒时间戳
2. `member` 用 `timestamp-nonce`，避免同毫秒覆盖

Lua 脚本一次完成 4 步：

1. 删除窗口外的旧请求
2. 统计窗口内请求数
3. 如果已达到阈值，计算 `retryAfterMs`
4. 如果还没超限，写入当前请求并刷新过期时间

因为这 4 步在 Lua 里原子执行，所以不会出现并发下“先查后写”的竞态。

---

## 4. 为什么滑动窗口比固定窗口更合适

固定窗口的问题在于它看的是“当前自然秒桶里有多少请求”，而滑动窗口看的是：

> **最近 1 秒内真实发生了多少次请求**

例如限制是“1 秒最多 5 次”：

1. 用户在 `900ms、920ms、940ms、960ms、980ms` 连续打 5 次，会全部通过。
2. 如果他在 `1010ms` 再打第 6 次，滑动窗口往前看 1 秒，前面那 5 次仍然都在窗口里，所以会被拒绝。

这就避免了固定窗口在秒边界的双倍放量问题。

---

## 5. 降级策略：Redis 异常时怎么办

这里我没有选择“Redis 一挂就全部放行”，因为这些都是关键写接口，完全放开会让登录、防刷、下单和支付入口瞬间失去保护。

当前的降级策略是：

1. 正常情况：走 Redis + Lua 滑动窗口。
2. Redis 不可用、脚本执行失败或返回异常结果：自动降级到本地固定窗口限流。

这个降级策略的取舍是：

1. 我接受降级期间“多实例口径不完全一致”。
2. 但我不接受关键写接口在 Redis 故障时完全无保护。

对应指标上，我把命中结果区分成：

1. `allow`
2. `blocked`
3. `degraded_allow`
4. `degraded_block`

这样上线后可以直接看出有没有频繁进入降级路径。

---

## 6. 关键实现原理

### 6.1 Filter 只负责三件事

`GatewayWriteOpsRateLimitFilter` 现在只负责：

1. 匹配需要限流的关键写接口
2. 解析限流身份（用户 ID / IP）
3. 根据限流决策决定放行还是抛出 429

真正的限流实现下沉到独立组件 `RedisLuaGatewayRateLimiter`，这样后续演进算法时不需要改网关过滤器主流程。

### 6.2 Redis + Lua 滑动窗口脚本

滑动窗口脚本的核心逻辑可以概括成：

```lua
redis.call('ZREMRANGEBYSCORE', key, '-inf', now - window)
local count = redis.call('ZCARD', key)
if count >= limit then
  return blocked_with_retry_after
end
redis.call('ZADD', key, now, member)
redis.call('PEXPIRE', key, window)
return allow
```

这里最关键的点是：

1. `ZREMRANGEBYSCORE` 先把窗口外历史请求清掉
2. `ZCARD` 统计的是“当前滑动窗口内真实剩余请求数”
3. 超限时用最早一条请求时间去算还要等多久

### 6.3 本地降级兜底

本地兜底仍然是固定窗口，但它已经不是主路径，只在 Redis 异常时启用。

它的作用不是追求最精确，而是保证：

1. Redis 抖动时服务还能继续提供有限保护
2. 关键写接口不会在异常时被完全打穿

---

## 7. 这版方案的取舍

### 优点

1. 比固定窗口更精确，解决秒边界突刺。
2. 比单机内存更适合多实例网关。
3. Lua 原子执行，能避免并发竞态。
4. 面试里容易讲清楚，逻辑直观。

### 代价

1. 每次请求要操作 Redis `ZSET`，比简单计数更重。
2. 窗口里请求越密，Redis 中间状态越多。
3. 如果未来把限流范围扩大到所有高频接口，这个方案成本会继续上升。

所以这版方案很适合当前阶段：

> 关键写接口不多，QPS 可控，优先解决固定窗口问题和多实例一致性问题。

---

## 8. 面试时怎么讲这次改造

### 30 秒版本

> 我把网关关键写接口限流从单机内存固定窗口升级成了 Redis + Lua 的滑动窗口限流。固定窗口的问题是秒边界会突刺，而且多实例下每台网关各算各的；滑动窗口会统计最近 1 秒内真实请求数，能避免临界点双倍放量。Lua 脚本会原子完成删旧记录、统计窗口和写入当前请求。为了避免 Redis 故障时关键写接口完全裸奔，我还保留了本地固定窗口作为降级兜底。

### 90 秒版本

> 这次我治理的是网关里的关键写接口，也就是登录、下单和支付。改造前的方案是本地内存固定窗口，问题有两个：一是秒边界临界突刺，二是多实例不共享计数。  
> 后来我把它改成了 Redis + Lua 的滑动窗口限流。每个接口和身份组合对应一个 Redis ZSET，score 是毫秒时间戳，member 是时间戳加 nonce。每次请求会通过 Lua 脚本原子完成 4 件事：删掉窗口外旧请求、统计当前窗口内请求数、超限时计算 retry-after、未超限时写入当前请求。  
> 登录按 IP 限流，下单和支付优先按用户 ID 限流。Redis 异常时，我没有直接全部放行，而是降级到本地固定窗口，避免关键写接口在故障期完全失去保护。这样做的结果是：主路径解决了固定窗口精度问题和多实例不一致问题，同时异常时还有兜底策略。

### 最该强调的三句话

1. 之前是单机固定窗口，后来改成了 Redis + Lua 的滑动窗口。
2. Lua 保证删旧记录、统计窗口和写入当前请求是原子操作。
3. Redis 异常时不是直接放行，而是降级到本地固定窗口兜底。
