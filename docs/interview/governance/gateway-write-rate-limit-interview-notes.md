# 网关关键写接口限流改造原理讲解（GCRA 终版）

## 1. 背景：为什么这块要连续演进两次

这次改造的是 `gateway-service` 里的关键写接口限流，主要覆盖：

1. 登录 `POST /api/user/auth/login`
2. 下单 `POST /api/v2/order/orders`
3. 支付 `POST /api/v2/order/orders/{orderNo}/pay`

这条链路一开始是**单机内存固定窗口计数**。它能挡住基础突发，但有两个明显问题：

1. **固定窗口临界突刺**：秒边界前后会出现双倍放量。
2. **多实例不一致**：状态只在网关本机内存里，扩容后整体阈值会被实例数放大。

所以我先做了第一轮升级，把它换成 **Redis + Lua 的滑动窗口**，先解决精度和多实例一致性问题。  
但在继续评估之后，我又做了第二轮升级，最终改成了：

> **Redis + Lua 的 GCRA 限流，Redis 异常时降级到本地固定窗口兜底。**

最终版选 GCRA 的原因不是滑动窗口不够用，而是：

1. 滑动窗口需要在 Redis 里维护窗口内请求日志，典型实现是 `ZSET`。
2. GCRA 只需要维护一个 `TAT`，也就是理论到达时间，Redis 状态更轻。
3. 对关键写接口这种持续运行的治理组件来说，GCRA 的长期维护成本更低。

---

## 2. 演进路径：固定窗口 -> 滑动窗口 -> GCRA

### 2.1 为什么先做滑动窗口

因为它是对固定窗口问题最直接的修复：

1. 固定窗口看的是“当前自然秒桶有多少请求”
2. 滑动窗口看的是“最近 1 秒真实发生了多少请求”

这一步的收益是立刻可见的：

1. 解决秒边界突刺
2. 状态搬到 Redis 后，多实例统一口径
3. 面试里可以明确说“我用 Redis + Lua 做了滑动窗口限流”

### 2.2 为什么终版又换成 GCRA

滑动窗口虽然精确，但代价是 Redis 要维护窗口内的请求记录。对 `1 秒 N 次` 这种限流，如果用 `ZSET` 记录每次请求时间，随着接口持续运行：

1. 每个 key 需要做 `ZREMRANGEBYSCORE`
2. 需要维护窗口内的请求日志
3. 状态量和请求密度直接相关

而 GCRA 的核心思想不是“窗口里有多少请求”，而是：

> **当前这个身份已经预支了多少未来时间额度**

所以 GCRA 在 Redis 里通常只需要维护一个值：

1. `TAT`，也就是 `Theoretical Arrival Time`
2. 可以理解成“按当前节奏，下一次理论上排到什么时候了”

这意味着：

1. 状态更轻
2. Redis 读写更简单
3. 更适合长期作为网关治理基础组件

## 3. 终版方案总览

### 3.1 限流维度

三个关键写接口不是同一维度：

1. 登录按 **IP** 限流
2. 下单优先按 **用户 ID** 限流，缺失时退回 IP
3. 支付优先按 **用户 ID** 限流，缺失时退回 IP

原因很简单：

1. 登录前没有稳定用户身份，只能按 IP
2. 已登录写接口按用户维度更公平，不容易误伤同 NAT 出口下的其他用户

### 3.2 Redis key 设计

每个“接口 + 身份”对应一个 Redis key，例如：

```text
gateway:rate-limit:login:ip:10.0.0.1
gateway:rate-limit:order_pay:uid:1001
```

和滑动窗口不同，GCRA 终版不再把每次请求都记录成一个 `ZSET` 成员，而是只维护一个 `TAT`。

---

## 4. GCRA 原理：它到底在算什么

GCRA 可以理解成“时间债务模型”。

假设规则是：

1. 1 秒最多 5 次请求
2. 那平均每 200ms 才应该来 1 次

GCRA 会维护一个理论时间点 `TAT`：

1. 如果请求来得太早，说明它在预支未来额度，应被拦截
2. 如果请求到达时间不早于允许阈值，就放行，并把 `TAT` 往后推进一个请求间隔

这套算法有两个核心参数：

1. **Emission Interval**：单个请求的理论间隔
2. **Burst Tolerance**：允许提前消费的最大时间额度，对应“可接受的突发能力”

对于 `1 秒 5 次`：

1. `Emission Interval = 200ms`
2. `Burst Tolerance = 800ms`

这样前 5 次可以瞬时通过，第 6 次必须等到理论时间追平一部分之后才能继续放行。

---

## 5. Redis + Lua 的 GCRA 实现

### 5.1 为什么还要 Lua

虽然 GCRA 状态比滑动窗口轻，但它依然需要 Redis 原子判断：

1. 读旧 `TAT`
2. 判断当前请求是否到得太早
3. 如果允许，写回新的 `TAT`
4. 如果不允许，返回 `retryAfterMs`

如果把这些步骤拆开在应用层做，就会有并发竞态。  
所以终版依然使用 **Redis + Lua**，只是脚本从 `ZSET` 滑动窗口变成了 `GET/SET` 型的 GCRA 脚本。

### 5.2 脚本逻辑

Lua 脚本核心流程可以概括成：

```lua
local tat = redis.call('GET', key)
if not tat then
  tat = now
end

local allow_at = tat - burst_tolerance
if now < allow_at then
  return blocked_with_retry_after
end

local new_tat = math.max(now, tat) + emission
redis.call('SET', key, new_tat, 'PX', ttl)
return allow
```

和滑动窗口最大的区别是：

1. 滑动窗口维护的是“过去窗口内发生了什么”
2. GCRA 维护的是“未来额度已经被排到哪里了”

---

## 6. 降级策略：Redis 异常时怎么办

这里我保留了和第一轮一样的降级原则：

1. 正常情况走 Redis + Lua GCRA
2. Redis 不可用、脚本失败或返回异常结果时，降级到本地固定窗口限流

我没有选择“Redis 一挂就完全放行”，因为这些都是关键写接口：

1. 登录入口失去保护，容易被撞库和刷接口
2. 下单、支付入口完全放开，治理层会瞬间失效

我的取舍是：

1. 降级期间接受精度下降和多实例口径回退
2. 但不接受关键写接口在故障时完全裸奔

---

## 7. 指标与可观测性

网关限流指标分成四类结果：

1. `allow`
2. `blocked`
3. `degraded_allow`
4. `degraded_block`

这样上线后可以直接回答三个问题：

1. 哪些接口真的在被限流
2. 有没有频繁进入 Redis 降级路径
3. 当前阈值是过松还是过紧

---

## 8. 这次终版方案的取舍

### 优点

1. 比固定窗口更平滑，没有秒边界临界突刺
2. 比本地内存更适合多实例网关统一治理
3. 比滑动窗口更轻状态，Redis 成本更低
4. Redis + Lua 原子执行，避免并发竞态

### 代价

1. 理解门槛比滑动窗口更高
2. 面试时需要能讲清楚 `TAT`、`Emission Interval` 和 `Burst Tolerance`
3. 要额外说明为什么它和令牌桶效果接近

所以我的对外表达通常是：

1. 先说明我从固定窗口升级到滑动窗口，解决精度问题
2. 再说明终版进一步演进成 GCRA，降低 Redis 状态成本

这样链路更自然，也更像真实工程演进。

### 30 秒版本

> 我治理的是网关关键写接口，也就是登录、下单和支付。最开始的实现是单机内存固定窗口，问题是秒边界会突刺，而且多实例计数不一致。后来我先把它升级成 Redis + Lua 的滑动窗口，解决精度和多实例统一问题；终版又进一步演进成 Redis + Lua 的 GCRA。GCRA 不再在 Redis 里维护窗口日志，而是只维护一个理论到达时间 TAT，状态更轻。Redis 异常时我没有直接放行，而是降级到本地固定窗口兜底。

### 90 秒版本

> 这次我治理的是网关关键写接口限流，覆盖登录、下单和支付。原始方案是单机内存固定窗口，缺点有两个：一是秒边界前后会出现临界突刺，二是多实例网关之间不共享计数。  
> 我第一步先用 Redis + Lua 把它改成滑动窗口，先把精度问题和多实例一致性问题解决掉；但滑动窗口要在 Redis 里维护窗口内请求日志，典型实现是 ZSET，长期运行状态成本会更高。所以终版我又把它演进成 Redis + Lua 的 GCRA。  
> GCRA 的核心不是存窗口日志，而是存一个理论到达时间 TAT。每次请求来了以后，脚本先读取旧 TAT，计算它是否比允许到达时间更早；如果太早就拒绝并返回 retry-after，如果允许就把 TAT 往后推进一个 emission interval。这样 Redis 里每个限流 key 只需要维护一个时间状态，比滑动窗口更轻。  
> 登录按 IP 限流，下单和支付优先按用户 ID。Redis 异常时，我没有完全放开，而是降级到本地固定窗口，避免关键写入口在治理组件故障时彻底失去保护。

### 最该强调的四句话

1. 原始方案是单机固定窗口，存在秒边界突刺和多实例不一致。
2. 第一轮升级是 Redis + Lua 滑动窗口，先解决精度问题。
3. 终版又演进成 Redis + Lua GCRA，只维护一个 `TAT`，Redis 状态更轻。
4. Redis 异常时降级到本地固定窗口，而不是直接全部放行。
