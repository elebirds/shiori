# 商品搜索 ES + CDC 面试问答

## 一、基础概念题

### Q1：为什么商品搜索要引入 Elasticsearch，而不是继续用 MySQL？

答：

因为这次需求已经不是普通列表筛选，而是全文搜索。

MySQL 更适合做结构化查询，但在下面几个点上不适合承担真正的搜索：

1. 中文分词能力弱。
2. 相关性排序能力弱。
3. 热门关键词会直接把压力打到交易主库。
4. 后续扩展搜索字段和权重时，维护成本会越来越高。

所以我把 MySQL 继续保留为真相源，把 Elasticsearch 用作搜索投影。

### Q2：你说的“搜索投影”是什么意思？

答：

搜索投影的意思是：

> Elasticsearch 里存的是面向搜索场景重组出来的一份副本，不是真相源。

这份副本只负责搜索，不负责交易正确性。真正的商品状态、库存和版本仍然以 `product-service` 的 MySQL 为准。

### Q3：什么叫 outbox？

答：

Outbox 模式的核心是把“业务写库”和“待发送事件”放进同一个本地事务里。

这样商品发布成功时，对应的搜索更新事件一定会写进 outbox 表，不会出现：

1. MySQL 已经成功
2. 但搜索消息没发出去

这种双写不一致。

### Q4：什么叫 CDC？

答：

CDC 是 `Change Data Capture`。这次用的是 Debezium 监听 MySQL binlog，把 outbox 表的变更抓出来，再写进 Kafka。

也就是说，这次不是应用代码自己直接发 Kafka，而是数据库变更被 CDC 异步采集。

### Q5：为什么分词在中文搜索里这么重要？

答：

因为中文不是天然用空格分词的。

如果没有分词，很多搜索只能退化成原始字符串包含判断，召回效果会很差。这里我在 ES 容器里装了 IK 插件，就是为了让商品标题和描述的中文检索更像真正的搜索。

### Q6：相关性排序是什么？

答：

相关性排序回答的是：

> 当多条文档都命中关键词时，哪条结果更应该排前面。

我这里默认让 ES 的 `_score` 优先，然后再用 `createdAt` 和 `productId` 做稳定排序。

## 二、架构设计题

### Q7：这次搜索链路的完整数据流是什么？

答：

完整链路是：

1. `product-service` 写商品事务
2. 同事务写 `p_outbox_event`
3. Debezium 监听 binlog
4. 把 outbox 变更发到 Kafka
5. `search-indexer-service` 消费 Kafka
6. 把搜索事件投影成 ES 文档

查询时则是：

1. 有 `keyword` 时优先查 ES
2. ES 失败时按配置回退 MySQL

### Q8：为什么不用业务代码直接双写 MySQL 和 ES？

答：

因为业务双写会把最难的一致性问题放到主写路径上。

它会立刻遇到：

1. MySQL 成功、ES 失败怎么办
2. 重试时会不会重复写
3. ES 短时不可用要不要把商品发布一起打挂

而 outbox + CDC 的思路是：

1. 先保证真相源和事件原子落库
2. 再让搜索索引异步构建

这样主写路径更稳，也更容易扩展下游。

### Q9：为什么不直接用 Debezium 同步商品主表到 ES？

答：

因为搜索系统不应该直接耦合交易主表结构。

我更希望 ES 消费的是“面向搜索的领域事件”，而不是原始表镜像。这样有几个好处：

1. 搜索字段由业务明确裁剪，不会被主表结构绑死。
2. 上架、下架、重建这些语义更清楚。
3. 后面如果搜索文档结构调整，不必直接暴露数据库内部设计。

### Q10：为什么要单独做一个 `search-indexer-service`？

答：

因为索引构建和商品交易写路径是两类完全不同的职责。

拆出来之后：

1. 搜索链路可以独立扩容。
2. ES 相关依赖不会污染主业务服务。
3. 索引失败不会把商品写请求一起拖垮。
4. 面试里能更清楚地讲“真相源服务”和“投影服务”的边界。

### Q11：为什么查询侧还要保留 MySQL fallback？

答：

因为搜索投影是增强能力，不是唯一依赖。

ES 异常时，如果直接把接口打挂，业务可用性会被搜索组件绑死。加上 fallback 后，虽然搜索体验会退化，但接口还能继续提供结果。

这个设计体现的是：

> 搜索体验优先，但服务可用性更优先。

### Q12：为什么无 `keyword` 的列表不走 ES？

答：

因为没有关键词时，本质上是普通分页列表，不是全文搜索。

这类请求原有 MySQL 路径已经足够清晰，而且过滤、排序规则本来就更贴近业务列表逻辑，没有必要硬切到 ES。

## 三、一致性与可恢复性题

### Q13：这套方案是一致的吗？

答：

是最终一致，不是强一致。

这里要分开说：

1. 商品真相源是 MySQL。
2. 搜索索引是异步投影。
3. 写库成功后，ES 允许有一个短暂传播延迟。

所以我不会夸成“强一致搜索”，而是会明确说这是“真相源强正确，搜索投影最终一致”。

### Q14：如果 CDC 或 Kafka 中间断了，会不会影响商品发布？

答：

不会直接影响商品发布，因为业务写路径只依赖 MySQL 和 outbox 本地事务。

链路断了的后果是：

1. 搜索索引停止推进
2. 但交易写库仍然正常

恢复后可以继续消费，必要时还可以全量重建。

### Q15：为什么必须做 `reindex`？

答：

因为任何搜索系统都不应该只依赖增量同步。

必须能全量重建，原因包括：

1. 新增索引字段
2. 改 mapping 或 analyzer
3. 历史消息丢失
4. 需要从真相源重新校正 ES

所以我补了内部 `reindex`，它会重新生成搜索 outbox，再走一遍整条 CDC 链路。

### Q16：为什么 `reindex` 不是直接脚本扫库写 ES？

答：

因为我希望重建和增量走同一条投影链路。

直接脚本写 ES 虽然快，但会带来两个问题：

1. 重建逻辑和增量逻辑分叉，后续容易不一致。
2. 很多搜索字段的构造规则会重复实现一遍。

而“重建时重新生成 outbox”能复用同一套事件模型和索引消费逻辑。

### Q17：ES 文档和业务库不一致时，你优先怎么修？

答：

优先用真相源重建，而不是手工修单条文档。

因为手工补单条通常只能修当前症状，不能证明整条链路恢复了。真正稳妥的做法是：

1. 确认链路恢复
2. 以真相源重新推导索引

## 四、查询与排序题

### Q18：你的默认排序为什么不是按创建时间，而是按 `_score`？

答：

因为有关键词的场景本质上是搜索，不是普通列表。

如果用户输入关键词，结果却还是主要按时间倒序，那体验更像列表页，不像搜索。所以默认应该先尊重相关性，再用时间和主键做稳定排序。

### Q19：如果用户显式选了价格排序，为什么还要保留 `_score`？

答：

因为显式排序表达的是主排序意图，但关键词相关性仍然有价值。

所以我的策略是：

1. 显式业务排序优先
2. `_score` 作为次级排序

这样既尊重用户排序意图，也避免结果完全丢掉搜索属性。

### Q20：你怎么处理中文分词？

答：

我在 ES 容器里安装了 IK 插件，让标题和描述字段能按中文语义分词。这样搜索“商品”“订单”这类中文关键词时，召回效果会明显好于原始字符串匹配。

### Q21：如果 ES 完全挂了，接口行为是什么？

答：

如果开启了 `mysql-fallback-enabled`，接口会回退到 MySQL 路径继续返回结果。

这意味着：

1. 服务不中断
2. 但搜索体验会退化
3. 分词和相关性排序能力会下降

这是一种典型的“能力降级而非服务不可用”设计。

## 五、深挖题

### Q22：为什么你说这套方案比“接个 ES 查询”更有技术含量？

答：

因为真正的难点不在“会查 ES”，而在“怎么把搜索系统做成一条可持续运维的链路”。

这次我补齐的是：

1. 真相源与搜索投影分离
2. outbox 解决事务边界
3. Debezium + Kafka 做异步增量索引
4. 独立 indexer 服务解耦职责
5. fallback 兜住查询可用性
6. reindex 兜住索引可恢复性

所以它不是单点技术，而是一整条搜索投影链路设计。

### Q23：如果面试官问“你这里是不是最终还是会有延迟”，怎么答？

答：

我会直接承认：

> 对，搜索索引是最终一致，不是零延迟强一致。

但我会继续补两句：

1. 这是主动选择，用写路径稳定性换取搜索投影异步构建。
2. 我通过 fallback 和 reindex 把最终一致的风险控制在可接受范围内。

这样答案会更成熟，而不是回避 trade-off。

### Q24：如果后面还要接推荐、风控、画像，你这套架构有什么优势？

答：

优势在于 outbox 已经把“业务写入”和“事件产出”稳定化了。

后面新增下游时，不需要再改主写事务去做一堆双写，只要：

1. 继续消费已有事件
2. 或者扩展事件模型

就能把新的投影或异步处理链路接出来。

### Q25：这次项目里你最希望面试官追问哪一点？

答：

我最希望他追问：

1. 为什么不用业务双写
2. 为什么要做 reindex
3. ES 挂了为什么还能兜底

因为这三点最能体现这不是“会用一个组件”，而是“能把搜索系统做成工程化方案”。
