# v0.7 路线图：支付微服务（余额托管 + CDK）

## 1. 背景与目标

当前交易链路中的支付仍是订单服务内的模拟逻辑。为与交易系统后续对接并支持资金域独立演进，v0.7 目标是引入独立支付微服务，先落地最小可用闭环：

1. 仅支持余额支付（不接真实第三方支付）。
2. 管理员可批量创建 CDK，用户可兑换为余额。
3. 订单支付进入托管，订单完结时再执行余额转移。
4. 支持幂等与失败补偿，保障资金一致性。

## 2. 范围

### 2.1 In Scope

1. 新增 `shiori-payment-service` 微服务（Java/Spring）。
2. 余额账户、资金流水、订单支付托管、CDK 批次与兑换能力。
3. 网关路由与鉴权规则新增支付域配置。
4. 订单 v2 支付改为调用支付服务余额托管。
5. 订单 `FINISHED` 时触发结算（托管转卖家）。
6. Docker Compose / Nacos / MySQL 初始化 / 监控抓取接入支付服务。

### 2.2 Out of Scope

1. 接入微信/支付宝等真实支付通道。
2. 前端管理台完整 CDK 页面和用户钱包 UI（v0.7 先后端就绪）。
3. v1 订单支付协议改造（保持兼容）。

## 3. 关键业务规则

1. 余额账户拆分为 `available`（可用）与 `frozen`（冻结）。
2. v2 支付流程：
   1. 买家支付时：`available -> frozen`（托管成功后订单进入 `PAID`）。
   2. 订单进入 `FINISHED` 时：买家 `frozen` 扣减并给卖家 `available` 增加。
3. 结算失败策略：阻塞完结，订单保持未完成状态。
4. CDK 规则：
   1. 单码一次性兑换；
   2. 支持批量生成；
   3. 可选过期时间；
   4. 仅保存哈希，不落明文。

## 4. API 设计

### 4.1 网关外部 API

1. 用户钱包
   1. `GET /api/v2/payment/wallet/balance`
2. 用户 CDK 兑换
   1. `POST /api/v2/payment/cdks/redeem`
3. 管理员 CDK 批次创建
   1. `POST /api/v2/admin/payments/cdks/batches`

### 4.2 服务内部 API（order -> payment）

1. 余额托管
   1. `POST /api/payment/internal/orders/{orderNo}/reserve`
2. 结算
   1. `POST /api/payment/internal/orders/{orderNo}/settle`
3. 释放托管（补偿）
   1. `POST /api/payment/internal/orders/{orderNo}/release`

## 5. 数据模型

### 5.1 payment 库

1. `p_wallet_account`
   1. 用户余额账户（可用/冻结）。
2. `p_wallet_ledger`
   1. 余额变更流水，记录前后状态与业务关联号。
3. `p_trade_payment`
   1. 订单托管记录（`RESERVED/SETTLED/RELEASED`）。
4. `p_cdk_batch`
   1. CDK 批次元数据（批次号、面额、数量、有效期）。
5. `p_cdk_code`
   1. 单码记录（哈希、状态、兑换人、兑换时间）。

### 5.2 order 库

1. `o_order` 新增 `payment_mode`：
   1. `SIMULATED`
   2. `BALANCE_ESCROW`

## 6. 兼容与迁移

1. `POST /api/v2/order/orders/{orderNo}/pay` 改为严格无请求体。
2. `POST /api/order/orders/{orderNo}/pay`（v1）保持原 `paymentNo` 协议不变。
3. 旧订单默认 `payment_mode=SIMULATED`，不会触发余额结算。

## 7. 一致性与幂等

1. 支付托管与结算均按 `orderNo` 幂等。
2. 订单端支付成功前若状态更新失败，调用 `release` 补偿。
3. 订单端完结流程中结算失败会回滚状态迁移。

## 8. 安全与鉴权

1. 网关新增支付域路由与权限码：
   1. `payment.wallet.read`
   2. `payment.cdk.redeem`
   3. `payment.cdk.manage`
2. 支付服务沿用网关签名校验与统一安全自动配置。

## 9. 验收用例

1. 管理员创建 CDK 批次成功，返回明文码，仅返回一次。
2. 用户兑换有效 CDK 后余额增加；重复兑换同一码失败。
3. 买家 v2 下单后支付，订单进入 `PAID`，买家余额冻结增加。
4. 订单 `FINISHED` 后冻结转卖家可用，资金总额守恒。
5. 结算异常时订单不应进入 `FINISHED`。

## 10. 风险与后续

1. v2 严格无请求体会影响旧前端调用，需要后续联调切换。
2. 当前先不提供对账任务，后续版本补齐每日资金核对与修复脚本。
3. 后续可扩展：
   1. 外部支付渠道适配层；
   2. 提现与退款子流程；
   3. 财务报表与运营后台。
