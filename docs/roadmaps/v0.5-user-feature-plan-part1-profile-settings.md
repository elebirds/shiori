# Shiori v0.5 用户功能完善计划（第一部分：资料拓展与个人设置）

## 1. 目标

v0.5 第一部分目标是把当前“仅昵称+头像 URL”的用户资料能力，升级为可用、可维护、可持续扩展的个人设置系统：

1. 完成用户基础资料字段拓展：头像、年龄、个人简介、性别。
2. 完成用户个人设置界面升级：信息分区、表单校验、上传体验、反馈提示。
3. 采用新版接口与交互基线（不以老客户端兼容为约束），并补齐回归测试。

## 1.1 执行状态（2026-03-04）

- [ ] M1：用户资料模型与 API 扩展
- [ ] M2：个人设置页重构与交互完善
- [ ] M3：头像上传闭环与质量收口

## 2. 范围

### In Scope

1. `u_user` 新增资料字段：`gender`、`birth_date`、`bio`（头像沿用 `avatar_url`）。
2. 用户资料查询/更新接口升级（按新版契约统一调整）。
3. 个人设置页升级（基础资料 + 安全设置双卡片保留）。
4. 新增头像上传流程（后端中转上传 OSS，前端不感知 OSS 连接信息）。
5. 补齐后端单测/MVC 测试、前端关键交互回归。

### Out of Scope

1. 用户隐私可见性策略（如“仅自己可见/好友可见”）。
2. 社交资料（院系、标签、社交链接等）。
3. 资料审核流（敏感词审核、头像审核）。

## 3. 数据与接口设计基线

## 3.1 字段策略

1. 性别：`gender TINYINT NOT NULL DEFAULT 0`（`0=UNKNOWN,1=MALE,2=FEMALE,9=PRIVATE`）。
2. 年龄：数据库存 `birth_date DATE NULL`，接口返回 `age`（按当前日期计算），避免“年龄”随时间失真。
3. 个人简介：`bio VARCHAR(280) NULL`，限制 280 字以内。
4. 头像：继续使用 `avatar_url VARCHAR(255)`，前端不再只依赖手动输入 URL。

## 3.2 API 约定（新版契约）

1. `GET /api/user/me`
   1. 返回新增字段：`gender`、`birthDate`、`age`、`bio`。
2. `PUT /api/user/me`
   1. 请求体字段按新版统一（`nickname`、`gender`、`birthDate`、`bio`、`avatarUrl`）。
3. `POST /api/user/media/avatar`
   1. `multipart/form-data` 上传头像文件（可携带裁剪参数）。
4. `GET /api/user/media/avatar/{avatarKey}`
   1. 由用户服务或网关代理返回头像资源，前端不直接访问 OSS 域名/预签名地址。

## 4. 里程碑

## M1：用户资料模型与 API 扩展

### 变更点

1. 新增 Flyway 迁移：为 `u_user` 增加 `gender`、`birth_date`、`bio` 字段。
2. 升级 `UserProfileRecord` / `UserProfileResponse` / `UpdateProfileRequest`。
3. 升级 `UserProfileMapper` 查询与更新 SQL。
4. 在 `UserProfileService` 增加业务校验：
   1. `birthDate` 不得晚于当前日期。
   2. 年龄计算结果需在合理区间（建议 13-120）。
   3. `bio` 允许空值，超长拒绝。
5. 增加用户域错误码（如资料参数非法、头像格式/大小不合法）。

### 验收

1. 新字段读写在 `/api/user/me` 闭环可用。
2. 非法参数返回稳定错误码与可读提示。
3. 头像上传接口能完成“上传并回填头像字段”的闭环。

## M3：头像上传闭环与质量收口

### 变更点

1. 用户端新增头像上传入口（文件选择 + 裁剪框 + 本地预览 + 上传进度）。
2. 前端上传仅调用用户服务接口，用户服务负责对接 OSS，成功后自动写入 `avatarUrl`。
3. 上传限制：
   1. 格式：`jpg/png/webp`
   2. 大小：建议 `<= 2MB`
4. 测试与文档：
   1. 后端：`UserProfileControllerMvcTest`、`UserProfileServiceTest` 覆盖新增字段与边界值。
   2. 前端：关键表单校验、裁剪流程、上传失败提示、成功回填。
   3. README 示例更新为新版用户资料接口与头像上传方式。

### 验收

1. 用户可裁剪并上传头像，随后在设置页立即看到回显。
2. 非法文件类型或超大小会被明确拦截。
3. 前端网络请求中不出现 OSS 预签名地址与 OSS 凭证信息。
4. 文档与实际接口一致，不再出现旧路径示例。

## 5. 风险与回滚

1. 风险：前后端字段不一致导致保存失败。
   1. 处理：按同一版本窗口联动发布前后端。
2. 风险：头像上传链路依赖对象存储配置。
   1. 处理：上传链路失败时回退默认头像，不暴露 OSS 细节到前端。
3. 风险：年龄规则过严引发误伤。
   1. 处理：参数区间可配置化，先宽后严。
4. 风险：头像裁剪组件兼容性问题影响移动端体验。
   1. 处理：预留原图上传兜底路径，裁剪能力通过开关控制启用。

回滚策略：

1. 前端可隐藏新字段，仅保留旧字段提交。
2. 头像裁剪能力可通过开关关闭，回退为原图上传。
3. 新增字段均为可空/有默认值，不影响旧数据读取。
