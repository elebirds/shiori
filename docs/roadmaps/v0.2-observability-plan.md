# Shiori v0.2 实施计划（稳定性与可观测性优先）

## 1. 目标

v0.2 不扩展新业务功能，优先把现有交易闭环升级为可量化、可观测、可回归：

1. 能快速判断系统当前是否健康。
2. 能定位链路抖动发生在哪一层（Gateway / Order / Notify）。
3. 能在 CI 中持续发现性能退化趋势。

## 2. 范围

### In Scope

1. Java 四服务与 notify 指标暴露标准化。
2. `deploy` 增加 Prometheus + Grafana 可选 profile。
3. 补齐 `perf/` 的 k6 资产与基线阈值。
4. CI 增加可观测性冒烟和性能基线任务。
5. runbook 增加 SLO 与排障流程。

### Out of Scope

1. 新业务能力（优惠券、离线通知持久化等）。
2. 分布式事务模型升级。
3. 分布式会话共享与全局推送一致性保证。

## 3. 里程碑

## M1：指标接入与最小观测闭环

### 变更点

1. `shiori-notify`
   1. 新增 `/metrics` 端点。
   2. 增加 WS 连接数、推送结果、MQ 消费结果、路由耗时指标。
2. `shiori-java` 四服务
   1. 统一开放 `health,info,prometheus` actuator 端点。
3. `shiori-order-service`
   1. 增加 outbox relay、超时消费、订单状态迁移业务指标。

### 验收

1. 5 个服务都可被 Prometheus 抓取。
2. 跑通一条下单支付链路后，order 与 notify 业务指标有增长。
3. 现有 smoke/e2e 回归通过。

## M2：部署观测栈（profile 化）

### 变更点

1. `deploy/docker-compose.yml` 新增 `obs` profile。
2. 新增 Prometheus 抓取配置。
3. 新增 Grafana datasource 和 dashboard provisioning。

### 验收

1. `docker compose --profile obs up -d` 成功。
2. Grafana 可直接查看核心链路面板。

## M3：压测基线与 CI 集成

### 变更点

1. 增加 `perf/k6-order.js`、`perf/k6-ws.js`、`perf/README.md`。
2. 增加 `scripts/ci/run_perf_baseline.sh`。
3. CI 新增 `perf-baseline` job（初期非阻塞）。

### 初始阈值

1. Order API：p95 < 300ms，错误率 < 1%。
2. WS 推送：p95 < 2s，超时率 < 1%。

## M4：文档与运行手册固化

### 变更点

1. `README.md` 增加 observability 与 perf 章节。
2. 新增/更新 runbook：指标释义、告警处理顺序、性能回归处置。

### 收口状态（2026-03-04）

1. 已完成 `README` 的双基线说明（`local-regression` / `perf-stress`）与参数矩阵。
2. 已完成 runbook 复盘与阈值治理：
   - `docs/runbooks/e2e-ci-perf-debugging-2026-03-04.md`
3. 已固化 `50013` 快速排查顺序与 `K6_*` 地址策略。

## 4. 风险与回滚

1. 指标高基数风险：限制标签维度，仅保留 `result/type/from/to`。
2. 抓取开销风险：初始 `scrape_interval` 使用 15s。
3. 阈值噪音风险：性能任务先非阻塞观察，稳定后再逐步收紧。

回滚策略：
1. 指标埋点为旁路，不改业务主路径，可按模块独立回滚。
2. 观测栈使用独立 profile，可整体关闭。

## 5. 完成定义（DoD）

1. 5 个服务指标可抓取且稳定。
2. Grafana 面板可开箱即用。
3. perf 资产可复现并写入 CI 产物。
4. README + runbook 与实现一致。
